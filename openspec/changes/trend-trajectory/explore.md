# Explore: Trend Trajectory

## Current State

Scores are stored **per briefing, per opportunity**. Every time the pipeline runs for a niche, a new `BriefingModel` row is created with a new set of `OpportunityModel` rows. Each `OpportunityModel` carries the full score decomposition (`trend_velocity`, `competition_gap`, `social_signal`, `monetization_intent`, `frustration_level`, `total`, `confidence`) and a `created_at` timestamp.

There is **no trajectory, delta, or history concept** anywhere in the domain, application, or infrastructure layers:

- `OpportunityScore` is a frozen dataclass — a pure point-in-time snapshot.
- `Opportunity` has no historical reference.
- `Briefing` aggregates opportunities but does not reference previous briefings.
- `BriefingRepository.get_latest()` only fetches the most recent briefing — older ones exist in the DB but are never queried.
- `OpportunityRepository.find_by_niche()` queries opportunities joined by `briefing_id` with no time window filtering.

In short: **history already accumulates passively** (every pipeline run creates a new briefing with its opportunities), but there is no logic to read, compare, or surface it.

Alembic uses `op.add_column` for additive changes — no destructive migrations yet. The convention is safe to extend.

### Pipeline execution model

`RunPipelineUseCase.execute()`:
1. Loads niche keywords
2. Collects signals from all `TrendDataPort` adapters (parallel, fault-tolerant)
3. Scores signals via `ScoringFactory.get_engine(discovery_mode)`
4. Enriches opportunities via `InsightPort.synthesize()`
5. Creates a new `Briefing` and persists it via `BriefingRepository.save()`

The pipeline is triggered on a cron schedule (`pipeline_schedule` in `Settings`, default: Monday 08:00 UTC). There is no deduplication or upsert — each run is a full independent snapshot.

## Key Questions

1. **Identity across runs**: How do we match "the same topic" between two briefings? Topics are free-text strings generated by an LLM. Exact string match will miss paraphrasing. Do we normalize? Lowercase + strip? Or accept exact match as good enough for V1?

2. **New table vs computed on-the-fly**: Do we store a pre-computed `ScoreHistoryModel` table, or do we query the last two briefings and compute the delta at read time?

3. **Retention policy**: The `opportunities` table grows linearly with every pipeline run. With weekly runs and ~20 topics per niche, that is ~1,000 rows/niche/year. Acceptable forever? Or prune rows older than N days?

4. **Trajectory definition**: Simple absolute delta (`total_now - total_7d_ago`)? Percentage change? Categorical label (`GROWING / STABLE / COOLING`)? Or all three?

5. **API exposure**: Does trajectory appear in the existing `GET /briefing/{niche_id}` response, or as a separate endpoint (e.g., `GET /niches/{niche_id}/trajectory`)?

6. **Domain ownership**: Should `Opportunity` know its trajectory, or should trajectory be a read-model concern (application layer only, never touching the domain entity)?

7. **Baseline window**: What is the comparison window — previous briefing (most recent before current), or fixed 7-day lookback? Fixed windows are more predictable; previous-briefing is simpler to implement.

### Key constraints discovered

- **SQLite in development** (`database_url` defaults to `sqlite+aiosqlite`). SQLite supports `LAG()` window functions since 3.25 (2018) — available but less battle-tested than PostgreSQL.
- **`BriefingRepository` has no `get_previous()` method** — any trajectory computation requires a new query or an extended port.
- **`OpportunityModel` has no `niche_id` foreign key** — niche is only reachable via `briefing_id → briefing.niche_id`. This means trajectory queries need a join through `briefings`.
- **Topics are LLM-generated strings** — exact string match is the only feasible identity strategy without semantic search infrastructure. Lowercasing is a safe normalization.
- **No `OpportunityId` is stable across runs** — each run generates new UUIDs. Identity must be by `(niche_id, normalized_topic)`.

## Hypothesis

The cleanest approach for V1 is to **compute trajectory at read time** from the existing `opportunities` data, with no new tables. The `BriefingRepository` port gains a `get_previous()` method; the application layer adds a `GetTrajectoryUseCase` (or enriches `GetBriefingUseCase`) that fetches the two most recent briefings for a niche, computes a `ScoreTrajectory` value object per matched topic (absolute delta + percentage + direction label), and returns an enriched response. A single additive Alembic migration adds an index on `(niche_id, generated_at)` via `briefings` for query performance. This approach requires zero schema changes to core tables, is reversible, and delivers the feature end-to-end in a single focused change.
